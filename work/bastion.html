Why a gateway

Direct SDK calls into a single LLM provider work fine — until they don't. When your primary provider has an outage, or their per-tenant rate limit trips at 3am, or a specific model version starts drifting, you need a layer that (a) fails over to another provider transparently, (b) circuit-breaks the sick provider before you burn latency retrying it, and (c) caches deterministic requests so you're not paying for the same completion twice.

Bastion is that layer. It speaks the OpenAI Chat Completions and Responses APIs on the client side, so any existing OpenAI SDK works unchanged — you point base_url at Bastion and it fans out across providers.

Ordered fallback

Each route defines an ordered list of provider/model pairs. A request tries them in order until one succeeds. The order is per-route (not global) so a "creative writing" route can prefer Claude, a "code completion" route can prefer GPT, and both share the same infrastructure.

route: "chat.default" providers: - openai / gpt-5 - anthropic / claude-opus-4-8 - bedrock / claude-opus-4-8 # regional fallback on_failure: - retry_next_provider - record_metric: provider.failure - circuit_breaker: 5xx or 429 → open

Circuit breakers

Each provider has an independent circuit breaker with three states:

  • Closed — normal operation. Failures increment a rolling counter.
  • Open — after N failures within window T, the breaker opens and all requests skip this provider entirely for a cooldown period.
  • Half-open — cooldown expired; the next request is a probe. Success → closes. Failure → back to open with a longer cooldown.

Defaults: N=5, T=30s, initial cooldown 15s with exponential backoff up to 5 minutes. All configurable per-provider.

Prefix cache

Deterministic requests (temperature = 0 and seed set) can be cached by content hash. Bastion computes a hash over (model, messages, temperature, seed, tool schemas) and looks it up in Redis before hitting a provider. Cache hits return in single-digit milliseconds and don't count against provider quota.

key = sha256(model + messages + temperature + seed + tools) ttl = 7d # configurable per-route on_hit: return cached response, no provider call on_miss: call provider, store response, return

On the reference workload (structured extraction, deterministic classifier calls), the cache hit rate is 71% — the same call rarely gets a fresh answer, and streaming isn't blocked because deterministic requests aren't streamed anyway.

Observability

Every request emits Prometheus metrics tagged with route, provider, model, status, tenant, cache_hit. Grafana dashboards ship with the repo:

  • Request rate + p50/p95/p99 latency per route
  • Per-provider success rate and circuit-breaker state
  • Cache hit rate + Redis memory usage
  • Cost estimate per tenant (tokens × provider rate)

Deploy

Kubernetes manifests, Helm chart, and Docker Compose file included. Horizontal scale is trivial — Bastion is stateless; Redis is the only shared state. Typical deployment is 2 replicas behind a load balancer with a single Redis for cache.

The point isn't that any single provider will fail. It's that all of them will, and you shouldn't be writing retry logic in twelve different services. — From the design note

When to use it

Bastion is the right starting point when: (a) you already have services calling an LLM SDK directly and want to add resilience without rewriting them, (b) a single provider outage would degrade a customer-facing surface, or (c) you have deterministic workloads (extraction, classification, evals) that would benefit from a shared cache.

It is not the right fit for: (a) single-tenant hobby projects where a 5-minute outage is fine, (b) streaming-only chat where the cache doesn't help, or (c) cases where you need provider-specific features that can't be normalized (e.g. Claude's computer-use tool).
