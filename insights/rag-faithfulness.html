Faithfulness is a testing discipline, not a prompt.

Most RAG systems ship with a hope-and-pray attitude toward hallucination. Here's how to turn citation faithfulness into a build-time gate — a per-commit evaluation harness that catches regressions before they reach production. With worked examples from a system running at 0.96 average faithfulness.

The problem

Every RAG system has the same failure mode: the retriever returns a chunk that looks relevant, the LLM generates an answer that reads confident, and neither the retriever nor the LLM noticed that the answer isn't actually in the retrieved chunk. The answer is plausible. It is not faithful.

Prompt engineering — "only answer using the provided context", "cite the exact passage", "say I don't know if unsure" — reduces the failure rate. It does not eliminate it. And here's the operational problem: you can't tell from the answer alone whether the retriever picked a bad chunk or the LLM ignored a good one. Both look the same to the user.

The solution isn't a better prompt. It's a test suite.

You can't tell from the answer whether the retriever misfired or the LLM hallucinated. Both look the same to the user. Only a test suite tells you which one is broken.

What faithfulness means, exactly

The term gets used loosely. Let's pin it:

Faithfulness is a per-claim property. For every atomic claim in the LLM's answer, the retrieved chunk it cites must actually support that claim. It's a binary yes/no per claim, aggregated to a score for the answer.

Faithfulness is not:

  • Answer relevance — does the answer address the question? (a different metric)
  • Retrieval precision — did the retriever return chunks about the right topic? (upstream of faithfulness)
  • Correctness — is the answer true in the world? (retrieval can be faithful to a chunk that is itself wrong)

Faithfulness is a local property: the claim is supported by the chunk. Whether the chunk is correct, whether the answer addresses the question, and whether the retrieval was good — those are separate metrics with separate tests.

Building the golden set

The heart of the harness is a golden set of question / expected-answer / expected-citation triples. Every merge to main runs the full pipeline against this set and computes faithfulness.

Building a good golden set is where most teams stall. Three rules I've held to across engagements:

Rule 1. Cover the failure modes you've seen

Every user-reported hallucination becomes a golden-set entry. This is the most important rule. If a claim was wrong once, it will be wrong again — either from the same chunk, or from a paraphrase of the same chunk, or from a completely different query that happens to trip the same weakness in the retriever.

Concretely: when a user flags an answer, capture the query, the retrieved chunks, the answer, and the actual correct answer. Add all four to the golden set. It becomes a regression test.

Rule 2. Cover the shape of the corpus, not just the topic

If your corpus has both concise-fact chunks (dates, numbers, definitions) and long-narrative chunks (explanations, procedures, arguments), your golden set needs entries from both. The retriever behaves differently on short chunks vs long ones. The LLM extracts differently from a definition vs a paragraph.

Rule 3. Include negative examples

Questions the corpus doesn't answer. The correct behavior is a refusal — "the provided context does not contain that information." Every third or fourth golden-set entry should be a negative. Otherwise your harness silently rewards over-confident answers.

Rule of thumb Start with 30-50 entries. It's manageable to hand-curate. You'll add another 5-10 per week from real user feedback. After six months you'll have 200-400 entries and the harness will catch regressions before humans notice them.

Scoring per claim

The atomic unit is the claim, not the answer. For each answer produced against the golden set:

  1. Split the answer into atomic claims. A separate LLM call does this — the prompt is "decompose this answer into a list of independent factual claims, each verifiable against a single source".
  2. For each claim, pair it with the retrieved chunk that the citation points to.
  3. Score the pair with the RAGAS faithfulness rubric — is the claim supported by the chunk? Yes / no / partial.
  4. Aggregate: faithfulness = supported_claims / total_claims.

Below the threshold — I typically set 0.90 as the CI-fail line — the answer is flagged. In production, flagged answers are shown to the user with a "reduced confidence" indicator rather than silently returned.

Watch out for the LLM-as-judge trap

You're using an LLM to score claims produced by an LLM. If both models share failure modes, you'll get high faithfulness scores on answers that are hallucinated in ways the judge model also hallucinates. Two guardrails:

  • Use a different model family for the judge than the generator. If Claude generates, use GPT to judge (or vice versa).
  • Calibrate the judge against human labels quarterly. Sample 30 pairs, have a human score them, compare to the judge. Judge-vs-human agreement should stay above 0.90. If it drifts, the judge model needs retraining.

The harness itself

The whole thing is one CI job. Here's the shape:

# .github/workflows/rag-eval.yml
name: RAG eval

on:
  push:
    branches: [main]
  pull_request:

jobs:
  faithfulness:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: python -m acuity.eval.run --golden data/golden.jsonl --out eval-report.json
      - run: python -m acuity.eval.gate --report eval-report.json --threshold 0.90
      - uses: actions/upload-artifact@v4
        with:
          name: eval-report-${{ github.sha }}
          path: eval-report.json

The run subcommand re-executes the full pipeline (retrieval + generation) against every golden entry and writes a per-claim JSON report keyed to the git SHA. The gate subcommand exits non-zero if the aggregate faithfulness dips below the threshold — the PR fails to merge.

Failure modes I've seen

Prompt change silently drops citations

Someone tightens the system prompt to make the answers shorter. The LLM interprets "shorter" as "drop the citations". Faithfulness stays at 0.96 because the remaining claims are still supported — but users can no longer verify anything. The harness didn't catch this because it scored only cited claims. Fix: add a citation-density metric — claims-with-citations divided by total-claims — as a separate gate.

Retriever version bump breaks a specific slice

New embedding model, aggregate faithfulness identical, but the "policy override" query slice tanks from 0.94 to 0.71. Aggregate score hid the regression. Fix: report faithfulness per slice (by query pattern, by document type, by intent) — not just a global number. Gate on the worst slice, not the mean.

Golden set drift

The corpus updates. Documents get revised. Old golden entries still assert claims against passages that no longer exist. Score drops mysteriously. Fix: golden-set entries carry an anchor to the specific chunk-hash they were authored against. If the anchor no longer exists, the entry is flagged for re-review rather than scored as a failure.

Cost budget

Faithfulness scoring is not free. For each answer under evaluation you make roughly one decompose-into-claims call and N score-per-claim calls. On a 200-entry golden set with average 4 claims per answer, that's ~1,000 judge-model calls per CI run.

The math: at Claude Opus rates in mid-2026, a full eval run costs $3-8. Running on every PR is $60-160 per month at a moderate merge cadence. Cheap insurance.

Wrap

Faithfulness is a testing discipline, not a prompt engineering exercise. Build the golden set from real user failures. Score per-claim, not per-answer. Use a different-family judge. Gate on the worst slice, not the mean. Report per git SHA. Fail the build.

Every prompt change becomes measurable. Every eval failure has a specific claim/chunk pair to inspect. And "we don't hallucinate" stops being an aspiration and becomes a build-time contract.
